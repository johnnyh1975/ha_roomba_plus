# Roomba+ v3.4.0 — Calendar, To-Do, and Coverage Analytics for lewis-Firmware Robots

> 3,803 tests passing · Config Entry version 25 (no migration needed) · HACS: update as usual

A native-platforms release: your cleaning schedule and maintenance tasks are now first-class Home Assistant entities instead of only sensor attributes. Alongside that, i/s/j-series robots on lewis firmware (22.52.10+) — which don't send local pose data over MQTT — finally get their coverage heatmap, stuck-hotspot detection, furniture-change alerts, and layout-change alerts populated from cloud data instead of staying permanently empty. Two lower-priority items from the original plan (a voice-command feature and one of four planned to-do items) were dropped after verification — see "Not in this release" below.

## ✨ New

### Cleaning schedule as a real calendar — `calendar.{name}_schedule`

Your robot's cleaning schedule (read from `cleanSchedule2` on i/s/j-series, legacy `cleanSchedule` on 900/600-series) now shows up as a genuine Home Assistant calendar entity — recurring events, visible in the Calendar dashboard, queryable by any automation that wants to know "is a cleaning scheduled in the next hour." Read-only. Always created, on every robot tier — an empty calendar just means no schedule is set yet, same as any other HA calendar with nothing on it.

One thing to know: iRobot's schedule data only carries a start time, never a planned duration, so each event shows a fixed 60-minute placeholder block rather than a guessed one — the event's description says so explicitly.

### Maintenance tasks as a real to-do list — `todo.{name}_maintenance`

Filter replacement and brush (or, on Braava, pad) cleaning now appear as Home Assistant to-do items with a real due date, computed from the same self-calibrated wear-rate estimate the existing sensors already use — check the countdown before you're surprised by "0 hours remaining." Marking an item done fires the exact same reset used by the existing Filter/Brush reset buttons, so your maintenance-life countdown restarts immediately either way you prefer to do it.

SMART-tier robots (i/s/j-series, Braava) also get a third item, **Reconfigure rooms**, whenever a Smart Map zone hasn't been given a name yet. Unlike the two maintenance items, this one resolves itself: once you name the zone through the existing naming wizard, it disappears from the list on its own — there's nothing to "complete" by hand.

*(A fourth item, a "time to retrain your Smart Map" reminder, was in the original plan but didn't make it — see "Not in this release" below.)*

### Coverage analytics now work on lewis-firmware robots too

If your i/s/j-series robot never populated the coverage heatmap, stuck-hotspot markers, furniture-change alerts, or layout-change alerts, this is very likely why: those all read from an internal occupancy grid that only ever gets fed by local MQTT pose data — and lewis firmware (22.52.10+) doesn't send any. This release adds a cloud-data path that fills the same grid from your mission history instead, on every cloud refresh. No setup needed; it activates automatically once the map alignment step (already shipped, silent, unrelated to this release) has bootstrapped itself, typically after a handful of missions.

Stuck-hotspot detection specifically depends on iRobot's cloud data actually containing stuck-event records for your robot — this is structurally wired up and tested, but whether it populates for a *real* stuck incident on lewis firmware hasn't been field-confirmed yet. If your stuck-hotspot map stays empty despite a robot that has genuinely gotten stuck, that's the open question — everything else (the heatmap itself, furniture-change, layout-change) doesn't depend on it and should work regardless.

## 🛠 Fixed

### Removing the integration now actually deletes its stored data

Found during this release's own documentation review: deleting Roomba+ (Settings → Devices & Services → Roomba+ → Delete) removed the config entry and its entities, but the 15 separate files it stores on disk per robot — mission history, coverage baselines, maintenance timers, robot profile, and the rest — silently stayed behind in Home Assistant's `.storage/` directory indefinitely. The documentation said removal was "clean"; it wasn't. Deletion now removes every one of those files too. If you want to keep your history before removing a robot, export it first — see [Replacing or selling your robot](README.md#replacing-or-selling-your-robot).

## 🔍 Under the hood

- `sensor.py` — a 5,800-line file holding every sensor — has been split into five smaller, domain-organized files. Pure internal reorganization: no sensor's entity ID, state, or attributes changed. Mentioned here only because if you've ever poked around the source, this is why the file layout looks different.
- A shared watermark now tracks, per robot, which missions have already been fed into the coverage grid — regardless of whether that happened via the existing local-MQTT path or the new cloud path above — so a real-pose robot never gets the same mission counted twice.
- Several defensive-parsing fixes found during pre-release testing (malformed schedule data, non-numeric cloud coordinates, unset configuration values) — none were ever observed in the field, all caught by hardening tests before shipping.

## 🚫 Not in this release

**Voice commands ("clean the kitchen", etc.) — dropped, not delayed.** While building this, we confirmed there's currently no supported way for a third-party Home Assistant integration to ship its own Assist voice sentences that work automatically after installation — Home Assistant's sentence system only loads from a file you'd have to create yourself in your own configuration directory, and the alternative (a fully custom voice agent) would replace Home Assistant's own built-in agent for your robot's device rather than adding to it. Neither is the "just works" experience this was meant to be, so it's cut rather than shipped half-working. We'll revisit if Home Assistant's platform changes.

**"Time to retrain your Smart Map" to-do item — dropped.** We couldn't find an existing, reliable signal for "your map layout has changed enough that a retrain would help" at the right granularity for a single to-do item — the closest existing detection fires per-furniture-item, not map-wide, and would have made the reminder more noise than signal.

## 📦 Upgrade notes

No breaking changes, no storage migration. `calendar.{name}_schedule` and `todo.{name}_maintenance` appear automatically after updating — no configuration required. The coverage-analytics improvements for lewis-firmware robots also require no setup; give it a few cloud refresh cycles after updating before expecting the heatmap/stuck-hotspot data to appear.
