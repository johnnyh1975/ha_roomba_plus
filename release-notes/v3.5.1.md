## 🛠 Fixed

### Clearer error when iRobot's own cloud certificate has problems

A real-world report (wecoyote5): iRobot's `disc-prod.iot.irobotapi.com` briefly had an expired TLS certificate on their end, and this integration surfaced it as a bare "unknown error occurred" with nothing pointing at what actually failed. Every cloud network call now catches this specific failure and explains clearly that it's almost always temporary and on iRobot's side, not something wrong with your setup — local vacuum control is unaffected either way.

This does not, and will not, include a way to skip certificate verification: that would remove protection against on-path interception for every future connection this integration makes, to work around a problem that's both temporary and outside the integration's control regardless.

### GS-SMART-UMF bootstrap never ran for robots with sparse initial UMF data

Real-world report (mdarocha, i3+): coverage map wasn't backfilling from cloud history the way it should for a pose-less robot. Root cause found by code review, independent of waiting for further field confirmation — the task that's supposed to construct a UMF aligner from cloud traversal evidence when the initial setup couldn't (no `points2d`/`regions` available yet) was gated on `umf_aligner is not None` at the dispatch site. That's backwards: a robot whose initial UMF response lacked the needed data would never get this rescue path scheduled at all, ever, no matter how much cloud data accumulated afterward. The function now constructs a fresh aligner itself when one doesn't exist yet, using whatever UMF data is available at the time it actually runs (which can be more complete than what was available at initial setup) — the dispatch gate no longer requires an aligner to already exist. Zero tests existed for this function before this fix; 6 new ones cover both the dispatch gate and the construction logic directly.

## 🔍 Under the hood

- `cloud_api.py`'s four network call sites (`_discover`, `_login_gigya`, `_login_irobot`, `_aws_get`) now share one helper (`_raise_clear_ssl_error`) that catches `aiohttp.ClientSSLError` specifically and re-raises as a `CloudApiError` with the explanation above. Non-SSL connection errors (timeouts, DNS, refused connections) are untouched and continue to propagate as before.
- 5 new tests in `test_cloud_api.py` for the SSL handling, including one that specifically confirms non-SSL errors are *not* swallowed by this change.
- `_async_bootstrap_umf_aligner` (callbacks.py) now handles `aligner is None` by constructing one, then falls through into the existing traversal-evidence refinement path if the fresh aligner still isn't confidently aligned — rather than treating construction and refinement as mutually exclusive. 6 new tests: 2 on the dispatch gate (`make_cloud_refresh_callback`), 4 directly on the function's construction logic.

## ✅ Resolved without a code change

**The `test_config_flow.py`/`test_rest980_migrate.py` test exclusion was stale, and it turns out CI never relied on it in the first place.** Both files were being skipped in this session's own local test runs for a documented reason (missing `homeassistant.helpers.service_info.dhcp`/`.zeroconf` modules in an earlier sandbox state) — checked directly, both now run cleanly (16 and 20 passed respectively), and the full suite runs clean with zero exclusions (3,926 passed). More importantly: neither `.github/workflows/validate.yml` nor `release.yml` ever excluded these two files — both just run `pytest tests/` against the whole directory. The exclusion was purely a local convention for this session's sandbox, never a real project requirement, so there's nothing to change in the repository itself — just something to stop doing in ad-hoc local runs going forward.

**No opt-out mechanism for the remaining ~9 Repair Issues.** Considered and deliberately not built. The original request for this came from feedback about too many things that weren't genuinely broken interrupting the user — which the redesign above already resolved at the root: 20 of 29 signals were removed or converted specifically because they failed a strict actionability test, not arbitrarily. The 9 that remain passed that same test — each one needs a person's attention and can't resolve itself. A toggle to silence a whole category of these would let someone permanently blind themselves to exactly the kind of thing this redesign was careful to keep visible, which just relocates the original problem rather than fixing it. Home Assistant's native per-issue dismiss/ignore already covers the legitimate need (silencing one specific instance you've already seen), which is a different and more appropriate granularity than a permanent category-wide switch. If a specific one of the 9 turns out to be genuinely over-triggering for someone in practice, that's a concrete, new report to evaluate on its own terms — not a reason to build a general-purpose escape hatch in advance.
