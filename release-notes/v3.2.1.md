# Roomba+ v3.2.1

> Config entry version: **24 → 25** (automatic migration, no action needed)
> Minimum Home Assistant: **2025.5.0**
> Breaking changes: **see "Upgrade notes" below — 900-series (EPHEMERAL) users only**

## Fixed

- **MQTT "stale" false alarm after recharge or stuck recovery.** The connectivity sensor briefly flagged a problem for a few seconds every time the robot resumed after a recharge or a stuck-recovery, even though MQTT was working fine.
- **Day summaries undercounted error missions.** Missions ending in a specific error code (e.g. `error_17`) were not counted as errors in daily summaries — they silently fell through to "no result".
- **Day summaries now prioritize stuck days over error days.** If a robot both got stuck and hit an error on the same day, the day is now correctly labeled by the more actionable signal (stuck).
- **Mission replay link missing on some cards.** Unified mission records were missing the internal mission counter needed for the replay feature; it's now included for both local and cloud-sourced records.
- **(900-series) Room outline drawn in the wrong place, and cut off in larger homes.** The outline shown on the live map could be offset from the actual cleaning path by several meters, and was invisible for any area outside a fixed 6×6 m window around the dock regardless of how much of the home had actually been mapped. The outline is now derived directly from the same coverage data used everywhere else, with no size limit and no offset.
- **(900-series) Phantom room could linger in room detection.** A room that had actually merged into a neighboring room (or been split between several neighbors) could keep showing up as its own separate room indefinitely. Both the single-neighbor and multi-neighbor cases are now detected and cleaned up automatically.
- **(900-series) Two doorways between the same two rooms could be merged into one.** In homes with two separate openings between the same pair of rooms, only one doorway was ever recorded — the two were averaged into a single, wrong position. Both are now kept as separate doorways when they're far enough apart.
- **(900-series) First room-outline snapshot after updating could be empty.** A timing issue meant the very first automatic backup of the room outline sometimes saved with no data. Fixed; later snapshots were never affected.
- **(900-series) Live map and room detection used swapped X/Y coordinates.** A confirmed bug (traced to the underlying `roombapy` library convention) meant the map's horizontal and vertical axes were swapped internally. This is now corrected — see "Upgrade notes" for what this means for your existing map data.
- **Current-room location entity missing on existing installations.** The underlying bug (an entity created disabled by default) was already fixed for new installs in a previous release, but existing installations never picked up the fix. A new migration re-enables the entity automatically for anyone affected.

## Changed

- **(900-series) Room-recognition data is no longer discarded after a stuck event.** Previously, once the robot got stuck, everything it cleaned for the rest of that mission was thrown away for room-detection purposes (the live map still showed it). Now that data is corrected against the dock's known position once the robot returns, and kept — instead of being discarded outright.
- **(900-series) Zone selection removed.** The room/zone selection dropdown and its "clean zone" button never had any real effect on 900-series robots — this hardware has no way to target a specific room or zone, and the button always just started a normal full clean regardless of what was selected. Removed to avoid suggesting a capability the hardware doesn't have. Smart-map robots (i/s/j/m-series) are unaffected — their zone selection is fully functional.

## Under the hood

Several new internal data stores start collecting in the background on 900-series robots (structure-only coverage data, room-boundary history, accepted position-jump log, last-10-missions trajectory log, and a periodic room-map safety snapshot). None of these change any current behavior or add a visible entity yet — they're groundwork for further room-detection improvements planned for a future release.

## Upgrade notes

**Config entry migration (24 → 25):** automatic, no action needed. Re-enables the current-room `device_tracker` entity for installations affected by the bug described above.

**900-series (EPHEMERAL) coordinate fix — please read if you have a 900-series robot:** the X/Y axis fix above changes how coordinates map to real-world directions for all *new* data going forward. Room-map data recorded *before* this update won't line up with data recorded *after* it. If your room map looks scrambled right after updating, this is why. There's currently no dedicated option to reset just the room-map data (removing and re-adding the integration does **not** clear it either), so the practical effect will fade out on its own as new missions outweigh the old. A proper reset option is being considered for a future release.
