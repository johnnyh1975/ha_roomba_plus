# Roomba+ v4.0.0a6

A mix of real field-reported bugs fixed, a new user-facing option, and the first real batch of
Prime feature-parity work (capability-aware entities, new diagnostic sensors) built on top of
this session's roombapy-prime work.

> 4,171 tests passing (full suite) · `roombapy-prime` pin updated to v0.1.11a19

## 🐛 Fixed: two real, field-reported bugs (chairstacker)

- **The Schedule calendar showed "Off" for the entire duration of an active, schedule-triggered
  mission.** Root cause was deeper than it looked: fetching occurrences from exactly `now` meant
  an already-started occurrence was pushed a full week ahead by the underlying occurrence-
  generation logic, so it was never even in the returned list to begin with — not just filtered
  out afterwards. Now fetches with a small backward margin and prioritizes a currently-ongoing
  occurrence over the next upcoming one.
- **`roomba_plus.smart_start` crashed for any Prime robot with no blocking sensors configured** —
  the no-blocking-sensors fallback path unconditionally called a Classic-only method that simply
  doesn't exist on a Prime robot. Fixed, and `BlockingManager` (start-blocking sensors) now
  actually works for Prime robots for the first time — previously it was never even instantiated
  for a Prime config entry. Room-targeted `smart_start` on a Prime robot now gives a clear,
  honest error instead of a crash or a misleading message — region-based cleaning commands
  remain unconfirmed for this generation, tracked separately.

## ⚙️ New: opt-out toggle for the schedule calendar entity

Directly in response to field feedback (chairstacker) that the calendar entity's forced sidebar
panel wasn't wanted. A new options-flow toggle (default **on**, so no existing installation loses
its calendar on upgrade) lets you turn the entity off entirely — not just hide it from the
sidebar. Available for both Classic and Prime robots; Prime's own options flow also got a proper,
tier-appropriate form for the first time (it used to silently show several Classic-only map/
correlation fields that meant nothing for a Prime device).

## 🧭 New: capability-aware Prime entities

Six existing Prime entities (mop pad/tank sensors, suction level, carpet boost, pad wash/dry
status) are now gated on the robot's own real, per-device hardware capabilities instead of being
created unconditionally regardless of what the specific robot can actually do — the same
principle Classic has used for years. Unknown capability (e.g. before the coordinator's first
successful fetch) still defaults to creating the entity; only a confirmed, explicit "not present"
suppresses it.

## 📊 New: eight additional Prime diagnostic sensors

All sourced from data that was already flowing through the integration but never exposed:

- Four lifetime mission-outcome sensors (total/successful/canceled/failed) — the exact same
  field family Classic's own equivalent sensors already use, just reached via the cloud instead
  of local MQTT
- Lifetime charge-cycle counts (successful/failed)
- System uptime
- Lifetime navigation-reset count
- Serial number (diagnostic)
- Dock error indicator (binary sensor) — no error condition has been observed yet on any real
  device, but the sensor is ready for when one occurs

All new sensors are diagnostic-category and off by default except the four mission-outcome
sensors, which mirror their Classic counterparts' default visibility.
