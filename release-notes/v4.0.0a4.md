# Roomba+ v4.0.0a4

> 4,062 tests passing (full suite, zero exclusions) · `roombapy-prime` pin updated to v0.1.11a16

## ✨ New: dock/pad status, suction level, and carpet boost for V4/Prime robots

Five new entities:

- **Dock status, pad wash status, pad dry status** — human-readable labels from a fully
  confirmed, 86-value enum (mostly error states; not translated into all 8 languages given how
  rarely most of them actually appear in practice)
- **Suction level** — a genuine `device_class=ENUM` sensor, 5 confirmed values, fully translated
- **Carpet boost switch** — reads and writes the account's real "boost suction when carpet is
  detected" setting (a genuine, sensor-driven iRobot feature, confirmed via their own public
  product documentation — not a three-way mode some app code suggested, which turned out to be
  dead code in the real app). The write mechanism itself is confirmed to work (the same one
  already confirmed for other shadow writes), but whether toggling it actually changes the
  robot's real carpet-boost behavior isn't confirmed yet the way start/stop/dock/find are.

## 🎉 Fixed: Locate ("find my robot") now actually works

The previous implementation used a REST endpoint that turned out not to trigger anything on a
real device. A multi-session investigation across two disproven mechanisms found the actual
working one: `send_simple_command("find")` — the same command channel already used for
start/pause/stop/resume/dock, just a different verb. Confirmed live (jayjay): a genuine, audible
chime with no robot movement.

## Also in this release

Dependency pin bumped to `roombapy-prime` v0.1.11a16 — two real UX fixes from field-tester
feedback: every credential prompt reworded from "Prime account" to "iRobot account" (avoiding
confusion with Amazon Prime), and `--blid` support for the same environment-variable pattern
already used for username/password/country-code. (The larger batch of work — four new staged
test packages, `DockState`/`ResolvedMissionStatus` completions, the `PolicyZoneFeature` bug fix —
shipped in v0.1.11a15, already picked up by the previous Roomba+ release.)

Also documented a real, structural parity gap: the real app's own "Start" button explicitly
fetches and sends the account's active cleaning preferences, something the basic start command
used here structurally cannot do — a plausible explanation for a mission always running at
unexpectedly high power with no adaptation.
