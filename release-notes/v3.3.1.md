# Roomba+ v3.3.1 — Card Unblock, Explain Fixes & Privacy Docs

> 3,565 tests passing · Config Entry version 25 (no migration needed) · HACS: update as usual

A focused patch release: five small, additive API/attribute additions the xiaomi-vacuum-map-card team needed for their next release, two foundation bugs found while building them, and a new README section on data privacy.

## ✨ New

### Temporal hotspot data — `format=hazards`

Stuck hotspot pins now carry `dominant_weekday`/`dominant_hour` when a clear pattern exists (e.g. "this spot jams mostly on Monday mornings") — sourced from the same pattern detector the Repair Issue already used internally. Pins with fewer than 8 recorded stuck events don't have enough data yet for a confident pattern and report `null` for both — expected, not a bug.

### Map overlay data — `image.{name}_map` / `image.{name}_rooms_map`

Both map entities gain three new attributes (aligned mode only), for anyone building a custom map overlay:

- `zones` — keep-out zones and robot-observed obstacle zones as raw polygons/points
- `door_markers` — inferred door-crossing positions accumulated across missions
- `furniture_candidates` — cells the FURNITURE detector has flagged as likely-moved furniture (the same signal behind `binary_sensor.*_layout_change_detected`, exposed here as a full list instead of first-candidate-only)

### Explain now works for cloud-source missions too

`GET .../mission/{mission_id}/explain` previously 404'd for any mission that only existed in your iRobot cloud history (no matching local record) — the "Why?" button on cards was silently unavailable for those rows. It now resolves cloud-only mission ids the same way local ones resolve. One honest limitation: two signals (`robot_lifted`, the `excessive_recharge` reason) depend on local-only telemetry that a pure cloud-only record never carries — they read as "no"/absent rather than a guess in that case.

### New README section: Data privacy & data flow

A straight answer to "what does this integration actually send where" — local MQTT vs. optional iRobot cloud, what's stored locally and where, and what (if anything) ever leaves your Home Assistant instance (short version: nothing, automatically).

## 🛠 Fixed

- **The `roomba_plus.explain_mission` service now also resolves cloud-only mission ids.** Found during a bug-hunting pass after the REST endpoint fix above: the service had the same gap the REST endpoint did, since it calls the same underlying logic — an id copied from `format=records` for a cloud-only mission would resolve via REST but still 404 through the service. Both now behave identically, as documented.
- **The cloud `missionId` (ULID) now actually resolves on the Explain endpoint.** Documentation had described this as working since v3.2.0; it never did — only the local record id matched. Both now work.
- **The Explain endpoint's documented response shape was wrong.** The REST API contract doc described fields (`anomaly`, `reason`, `signals`) that don't exist in the actual response. Corrected to match the real shape (`is_anomalous`, `anomaly_reason`, `robot_lifted`, `error_code`, `recommended_action`) that's been returned all along — no behavior change, documentation only.

## 📦 Upgrade notes

No breaking changes, no storage migration. All additions are purely additive — existing attributes/fields are unchanged. Card users: update to xiaomi-vacuum-map-card v2.3.0+ (once released) to make use of the new map overlay data and cloud-source Explain support.
