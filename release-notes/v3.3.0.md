# Roomba+ v3.3.0 — Room Rhythms, Mission Maps & a Foundation Fix

> 3,510 tests passing · Config Entry version 25 (no migration needed) · HACS: update as usual

This release teaches Roomba+ *when* each room needs cleaning, lets one service call act on it with a travel-optimized route, and brings the official app's "Mission Cleaning Map" into Home Assistant — for every mission in your history. It also fixes a shipped bug that silently emptied two existing features on live installations.

## ✨ Highlights

### Per-room cleaning rhythms — `sensor.*_rooms_overdue` *(SMART + cloud)*

Your robot now knows which rooms are due. Each room's expected interval is **learned from its own cleaning history** — no configuration needed. Want a hard rule instead? Open the integration options → *Per-room cleaning frequency* and pick Daily / Every 2 days / 3× per week / Weekly per room; an explicit choice always beats the learned interval.

The sensor's state is simply the number of overdue rooms (perfect for automations and badges). Attributes show every room with its days-since-last-clean, expected interval, and whether that expectation is configured or learned — plus a fully self-calibrated **suggested interval per room** derived from how fast each room actually re-dirties, and a `daily_suggested` list for rooms that would benefit from daily cleaning.

### One-call catch-up cleaning — `roomba_plus.clean_overdue_rooms`

Fire it daily from an automation without any guards: it cleans exactly the rooms that are due, worst-overdue first, **with the route between the selected rooms travel-optimized starting from the dock** — no more criss-crossing the home. When nothing is due, it does nothing (logged, no error).

Its sibling `roomba_plus.auto_clean_dirty_rooms` targets the rooms that are currently **dirtier than your household average** (learned per-room dirt index), same route optimization, with a whole-house fallback while the index is still learning (fewer than 10 recorded cleanings per room).

### Mission cleaning maps *(SMART + cloud)*

Every finished mission's **real coverage** — the points the robot actually cleaned, including the gaps — is now available:

```
/api/roomba_plus/{entry_id}/missions/latest/map.png
/api/roomba_plus/{entry_id}/missions/{record_id}/map.json
```

The PNG works directly in a picture card or a "mission finished" notification. The JSON carries raw coordinates plus your room outlines — this is also the data source the upcoming card release will use for interactive coverage replay. Responses are verified against the mission's own map header (you will never silently get the wrong mission's map) and cached for 24 h.

*Note for i-series (i7/i8) owners: coverage layers are field-confirmed on j-series firmware; i-series confirmation is still pending. If your robot's missions return "no coverage layer", that's the known open question — a field report would be very welcome.*

### Dirt ↔ sensor correlation *(opt-in, fully local)*

Ever suspected the robot collects more dirt on humid days, or when pollen is high? Pick any HA sensors in the integration settings (*Correlate dirt with sensors*); Roomba+ snapshots them at every mission start and, once 30 missions are paired, computes the correlation **entirely on your machine** — nothing leaves Home Assistant. A new diagnostic sensor reports the strongest correlation, and a Repair issue tells you plainly when a strong one is found ("collects significantly more dirt when *Humidity* is high, r=+0.61").

## 🛠 Fixed

- **Room history features now work on live installations.** `sensor.*_room_cleaning_history` (v3.1.0) and the coverage-frequency REST endpoint (v3.2.0) read a record field that only *imported* histories ever carried — on normally-running installations both were silently empty. Room names are now derived from the mission timeline itself; imported histories keep working. If these features always showed nothing for you: that was this bug.
- **Explicit `null` values from the robot no longer crash entity updates.** Ten unguarded reads in the vacuum platform (`pose`, `cleanMissionStatus`, `padWetness` — a known Braava firmware behavior — `bin`, `mopReady`, and the command chain) now handle explicit nulls; ten regression tests lock this in.
- **Transient cloud hiccups no longer flap entities to unavailable.** Network errors and timeouts now benefit from the same grace period that HTTP errors already had — a Wi-Fi blip returns the last good data instead of marking everything unavailable.
- **Mission timer writes are batched.** The per-message disk write during missions (every few seconds, for the whole mission) is now debounced to one write per 15 s — kinder to SD cards; pending writes flush on shutdown and reload. Trade-off: up to 15 s of elapsed-time may be lost on a hard power cut.
- Degraded cloud responses (non-JSON bodies, incomplete credentials, malformed coordinates) are handled with clear errors instead of stack traces.

## 🔍 Under the hood

- Store internals are now behind public APIs with an AST-based guard test — groundwork for safer future refactors.
- A thread-safety fix in the map-alignment bootstrap path (local-build-then-rebind) removes a rare crash window on the MQTT render thread.
- Research groundwork: room-type verification logging (flag-file gated, silent by default) ships in this release to answer an open documentation question with real field data.

## 📦 Upgrade notes

No breaking changes. No storage migration. New features are additive; the correlation feature is opt-in and off by default. EPHEMERAL-tier robots (900-series): the room-rhythm features require per-room mission data that only SMART robots provide — see the README's tier table.
