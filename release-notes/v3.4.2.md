# Roomba+ v3.4.2 — Notification Blueprint and a Robustness Sweep

> 3,878 tests passing (+40 since v3.4.1) · Config Entry version 25 (no migration needed) · HACS: update as usual

A patch release with one small new feature and a genuine robustness sweep: 25 defensive-parsing fixes across 10 files, several of them in code paths — the base entity class, the cloud coordinator's main update loop, the main vacuum entity — where a single malformed cloud response could previously have taken down more than intended. None of these were ever reported by a user; all were found through a systematic review after a smaller fix turned out not to be isolated.

## ✨ New

### Curated notification blueprint

A ready-made automation blueprint covering five notifications most people actually want: mission complete, maintenance due, robot stuck, map retrain detected, and battery capacity critical. Each is individually toggleable, and you plug in your own notification action (mobile app, `persistent_notification`, TTS, whatever you already use) rather than being locked into one delivery method.

The battery-critical trigger deliberately doesn't use a fixed capacity percentage — it watches `sensor.*_estimated_battery_eol`, which already learns your robot's own degradation rate, so the same warning buffer (default: 14 days) means something similar regardless of your robot's age or usage pattern.

Import it directly — see [Automations → Notification blueprint](docs/AUTOMATIONS.md#notification-blueprint) for the one-click badge and setup steps. One import covers all your robots; create one automation instance per robot.

## 🛠 Fixed

### A null-guard gap, found in one place and then found to be systemic

While investigating a community-reported favorites issue, a defensive-parsing gap turned up in `select.py`: `pmap.get("active_pmapv_details", {})` only guards against that key being *missing* — not against the cloud API returning it with an explicit `null` value, which it demonstrably does. That pattern alone would have been a small fix. Checking whether it was isolated turned into a project-wide sweep, and it wasn't:

- **6 more instances in `cloud_coordinator.py`**, including one unguarded directly in the main `_async_update_data` loop — a null response there could have prevented the integration from completing setup on first load for an affected robot, not just degraded one feature.
- **3 instances in `entity.py`** — the base class every single entity in this integration is built on. A `null` `hwPartsRev` or `dock` value would have broken entity construction for the whole robot, not just one entity.
- **9 more instances** across `device_tracker.py`, `presence_manager.py`, `sensor_rooms.py`, `dirt_threshold_manager.py`, `sensor_helpers.py`, `todo.py`, `button.py`, `binary_sensor.py`, and `mission_store.py` (the last one feeding the main vacuum entity's `planned_room_order` and `mission_destination` attributes).

All 24 follow the same fix (`(dict.get(key) or {})` instead of `dict.get(key, {})`), all now covered by regression tests — including a new `test_entity.py`, since the base entity class had no dedicated test file before this release.

### Favorites with multiple command steps no longer fail silently

If a favorite's `commanddefs` ever contains more than one entry — confirmed as a real, supported shape in iRobot's own data model, in both the Classic and Prime apps, though we couldn't confirm from static analysis alone whether real-world favorites ever populate it with more than one — `FavoriteButton` now logs a warning naming exactly how many entries were dropped, instead of silently acting on only the first one. This is a defensive fix, not a full resolution: the underlying question (issue #9) remains open pending real field data.

### SKU diagnostics for product families we don't have hardware data for yet

`get_robot_profile()` previously returned silently with no profile for any SKU prefix it didn't recognize. It now distinguishes two cases in the log: a genuinely unknown prefix (`DEBUG`), versus a prefix confirmed to belong to a real iRobot product family we simply haven't profiled yet — Combo (`c`), and several newer families (`a`, `p`, `q`, `t`, `y`) confirmed to exist via iRobot's own public configuration data (`INFO`). No new hardware assumptions were added — this only makes it obvious, in a field tester's log, when their device belongs to a family this integration doesn't have calibration data for yet.

### Mission Map now works for cloud-only-source rows too

`GET /missions/{record_id}/map.json` and `.png` previously 404'd on any mission whose record only ever existed in your cloud history and never made it into local storage — the same synthetic `"c_{ts}"` id format `/mission/{id}/explain` already learned to resolve back in v3.3.1, but that fix was never carried over here. Both endpoints now accept `c_{ts}` ids the same way `/explain` does. A cloud-only record with no `pmaps_info` (EPHEMERAL tier, or recorded before the cloud merge existed) still 404s afterward — same as it always has for local records in that state; this fix only closes the "never even looked" gap, not the "no coverage data exists" one.

## 🔍 Under the hood

- A project-wide `pyflakes` pass removed 21 genuinely dead imports and 2 unused local variables (both a leftover `n = len(polygon)` in an already-simplified ray-casting helper, duplicated in `grid_store.py` and `umf_aligner.py`). Checked individually against re-export usage first — `sensor.py`'s ~58 "unused" imports are a deliberate, actively-used re-export pattern for test helpers and were correctly left alone.
- `strings.json` was missing `entity.calendar.schedule.name` and `entity.todo.maintenance.name` entirely — stale since v3.4.0 added those platforms, even though every translation file (including this release's freshly-reverified `pl.json`, contributed by **mdarocha**) already had them. Added, so the schema source and the shipped translations agree.
- `select.py`'s and `cloud_coordinator.py`'s test suites gained regression coverage for every one of the null-guard fixes above; several were verified to actually catch the regression by re-introducing the bug against the new test before re-fixing it.
- `quality_scale.yaml`'s test-coverage comment was corrected — it referenced a `tests_integration/` directory and separate `pytest.ini` that no longer exist in the current tree, and had a stale test count.
- README's version badge and test count, both stale since v3.4.0, corrected.
