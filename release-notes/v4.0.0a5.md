# Roomba+ v4.0.0a5

The biggest batch of real, field-confirmed fixes this project has shipped — three previously
silent bugs affecting every V4/Prime install, found through a systematic architecture review
prompted directly by real field reports.

> 4,121 tests passing (full suite, zero exclusions) · `roombapy-prime` pin updated to v0.1.11a18

## 🐛 Fixed: three bugs affecting EVERY Prime install since the very first release

- **The live cleaning map (`PrimeMapImage`) had never been reachable at all** since v4.0.0a0 —
  `Platform.IMAGE` was simply never in the list of platforms forwarded for Prime robots, despite
  fully working, unit-tested code existing for it. Fixed.
- **Every Prime robot's own device page** (Settings → Devices) showed a generic "Roomba XXXX"
  name, no model, no serial number, and no firmware version — despite individual sensors already
  showing the same data correctly. The device-info code path had simply never been built for
  Prime at all. Now shows the real robot name (no migration needed — it was always captured at
  onboarding, just never used here), model/serial (from a new, best-effort account lookup), and
  firmware version.
- **Home Assistant's own "Download diagnostics" button crashed immediately** for every Prime
  robot — it unconditionally touched an object that doesn't exist for cloud-only devices. Fixed
  with a proper, genuinely useful Prime-specific diagnostics report instead (connection health,
  coordinator status, which shadows are seeded).

## 🐛 Fixed: two real, field-confirmed sensor bugs (chairstacker)

- **Sensors intermittently showing "Unknown"**, traced to a partial cloud update overwriting the
  entire known state instead of merging into it — a small, routine update (e.g. mission status
  changing) was wiping out battery, dock status, and everything else at once.
- **Battery specifically still went to "Unknown" right at mission start**, even with the fix
  above — a second, more subtle version of the same underlying issue: an update can include a
  field as an *explicit* null rather than just omitting it, and the first fix still let that null
  overwrite a real, already-known value. Now it can't — new information always wins, but null is
  never treated as new information capable of clobbering a real one.

## 🛠️ Fixed: onboarding routing a Prime robot into a broken setup path

Prime robots respond to the same local network discovery broadcast Classic robots use, even
though they can't be set up that way at all — causing them to appear as a misleading "local IP"
option during onboarding, and in the worst case, to be silently routed into that broken setup
path automatically with no choice offered. Both fixed.

## 🧱 New: three structural safeguards against this whole class of bug

Built directly in response to the platform-forwarding and device-info bugs above, so the same
mistake can't recur silently a third time:

- A comprehensive test that actually calls every Prime platform's own setup function against a
  realistic config entry and confirms it creates real entities — in both directions (does
  everything *listed* actually work, and does everything that *should* be listed actually appear
  in the list).
- An automated check that the `roombapy-prime` version pin agrees across `manifest.json` and
  both CI workflow files — these have drifted out of sync more than once.
- The locale-slug naming-convention guard now discovers every platform file automatically
  instead of relying on a manually maintained list — the exact kind of list that caused two of
  the gaps above.

## ✨ New: Prime schedule calendar

Read-only `calendar.roomba_*_schedule`, matching Classic's own pattern — shows each schedule's
own room/zone (resolved to a real name via the account's map data). Weekly-recurring schedules
only; bi-weekly/monthly/one-time are deliberately skipped rather than shown with a guessed date.
Classic's own calendar also gained the same room/zone display, differentiated correctly between
SMART-tier (has real room data) and EPHEMERAL-tier (no room concept at all) robots.

## Also in this release

- Two more background-task consistency fixes: the live map's own watch loop now has the same
  defensive retry behavior already added to both coordinators, and uses the same
  config-entry-scoped background task mechanism as everything else in this project (was a bare
  `asyncio.create_task()` before).
- `--blid` and every credential prompt across all `roombapy-prime` scripts got real UX fixes from
  field-tester feedback (see that project's own v0.1.11a16 release notes).
- `verify-region-commands` gained a `--send-with-initiator` stage, and its own stage-2 crash
  (against every real favorite, not just one) is fixed — see `roombapy-prime`'s v0.1.11a17
  release notes for the full detail.
- New building blocks in `roombapy-prime` for account/room-name resolution
  (`get_household_id()`, `build_room_name_map()`) — groundwork for the schedule calendar above.
