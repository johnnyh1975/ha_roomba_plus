# Roomba+ v4.0.0a3

> 4,050 tests passing (full suite, zero exclusions) · `roombapy-prime` pin updated to v0.1.11a15

## 🐛 Fixed: a real safety gap in how Prime robots are recognized

The previous release (v4.0.0a2) added a second recognized SKU ("N", alongside "G") using a
single-LETTER prefix check. Decompiled source (`com/irobot/core/SkuUtils.java`) has now shown
this was a real, not just incomplete, risk: iRobot reuses some letters — "R" and "Q" specifically
— across **both** generations. `R980020` (a Roomba 980, Classic) and `R285020`
(EnhancedComboNextPlus, cloud-only Prime) share the same first letter but are completely
different robots.

A single-letter check would eventually have misclassified a real Classic robot as Prime —
skipping its local network setup entirely and silently treating it as cloud-only. That's a worse
failure mode than the original bug this project fixed in v4.0.0a2 (which at least failed loudly
with a reportable "not found on network" error).

Fixed: SKU recognition now matches on the first **three** characters (letter + 2 digits), which
correctly separates every known case, including the newly-found lineup of Prime-generation
platforms (EssentialCombo/EnhancedCombo/MaxCombo family and their variants). Two real field
devices (`G185020`, `N185240`) remain confirmed; the rest of the newly-identified prefixes are
sourced directly from iRobot's own app code, not yet individually field-confirmed.

## ✨ New: live cleaning map for V4/Prime robots

A new image entity shows the robot's live cleaning map — the occupancy grid it builds while
cleaning, updated as new data arrives over the cloud connection. This uses
`roombapy-prime`'s `decode_rawmap_to_png()`, confirmed against a real device's actual map data
and visually verified to match the app's own view.

This first version is intentionally minimal: it's the occupancy grid image itself, not a full
map card. No pose trail, no room-outline overlay, no coverage heatmap yet — those are
possibilities for later, once real usage confirms this basic version works well across
different homes and robots.

## ✨ New: battery, bin, tank, pad, runtime, firmware, and connectivity sensors

Seven new entities, all reading real, live-confirmed values (not placeholders):

- **Battery percentage** — same device class and unit as the Classic battery sensor
- **Bin present** / **Mop tank present** — binary sensors, same device class as their Classic
  counterparts
- **Detected pad** — which mop pad type the robot currently reports
- **Lifetime runtime hours** — with minutes as an attribute
- **Firmware version** — confirmed a plain string via Ghidra decompilation of the app's own code
- **Robot connectivity** (binary sensor) — the robot's *own* reported connection to AWS IoT.
  Deliberately separate from the existing "Connection health" sensor, which reflects this
  integration's own connection instead — the two could legitimately disagree.

This closes out the battery-status search this project has been running for weeks: the data
lives in previously-unknown, read-only named shadows, not in the classic robot state at all.

A new coordinator seeds all eight named shadows once at startup (so sensors have a real value
immediately), then listens for live updates via a new subscription method
(`watch_named_shadows_updates()`, from `roombapy-prime`). That specific push mechanism is
reasoned and safety-checked but not yet confirmed live-tested — if it turns out not to deliver
updates in practice, sensors simply keep their last-known value rather than breaking, so this is
safe to try regardless.

## 🎉 Fixed: Locate ("find my robot") now actually works

The previous implementation used a REST endpoint (`poll_echo_value()`) that turned out not to
trigger anything on a real device. A multi-session investigation across two disproven mechanisms
found the actual working one: `send_simple_command("find")` — the same command channel already
used for start/pause/stop/resume/dock, just a different verb. Confirmed live (jayjay): a genuine,
audible chime with no robot movement.

## ✨ New: dock/pad status, suction level, and carpet boost for V4/Prime robots

Five new entities:

- **Dock status, pad wash status, pad dry status** — human-readable labels from a fully
  confirmed, 86-value enum (mostly error states; not translated into all 8 languages given how
  rarely most of them actually appear in practice)
- **Suction level** — a genuine `device_class=ENUM` sensor, 5 confirmed values, fully translated
- **Carpet boost switch** — reads and writes the account's real "boost suction when carpet is
  detected" setting (a genuine, sensor-driven iRobot feature, confirmed via their own public
  product documentation — not a three-way mode some app code suggested, which turned out to be
  dead code in the real app). The write mechanism itself is confirmed to work (the same one
  already confirmed for other shadow writes), but whether toggling it actually changes the
  robot's real carpet-boost behavior isn't confirmed yet the way start/stop/dock/find are.

## Also in this release

Dependency pin bumped to `roombapy-prime` v0.1.11a15, which brings a large batch of work: four
new staged test packages for previously-untested write paths (region-aware mission commands,
schedule writes, favorite writes, virtual wall/zone writes), `DockState` (86 values) and
`ResolvedMissionStatus` (49 values) fully completed from partial, a real parsing bug found and
fixed (`PolicyZoneFeature` mis-parsing virtual-wall geometry), typed
`ConnectionStatusShadow`/`SoftwareStatusShadow` fields, and `decode_rawmap_to_png()` with the
`[map]` extra for the live map above. Also documented a real, structural parity gap: the real
app's own "Start" button explicitly fetches and sends the account's active cleaning preferences,
something the basic start command used here structurally cannot do — a plausible explanation for
a mission always running at unexpectedly high power with no adaptation.
