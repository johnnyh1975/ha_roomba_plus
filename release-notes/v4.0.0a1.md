# Roomba+ v4.0.0a1

> 4,037 tests passing (full suite, zero exclusions) · `roombapy-prime` pin updated from v0.1.11a6 → v0.1.11a12 · Config Entry version unchanged (no migration needed)
>
> This alpha is primarily a stability and dependency-freshness release, prompted directly by a
> real field report of lost connectivity that a full Home Assistant restart couldn't recover
> from. Battery percentage and a direct docked/charging boolean remain unexposed as entities in
> this alpha — the underlying data location has been found upstream (see below), but isn't wired
> into a sensor yet.

## 🐛 Fixed: a real, permanently-stuck-connection bug

A field tester's Prime robot lost MQTT connectivity and stayed lost across multiple full HA
restarts — Connection health kept showing "ok" while Mission event silently went stale, then
"Unavailable". Investigating this found two things:

- **This integration's own `roombapy-prime` dependency had been pinned to v0.1.11a6 this whole
  time** — `manifest.json` was never updated past the very first alpha's pin, meaning every fix
  since (six releases' worth) was sitting unused. Now correctly pinned to v0.1.11a12.
- **A real reconnect weakness in `roombapy-prime` itself, now fixed upstream**: if the live
  connection dropped after its login token had already expired, reconnection could get stuck
  retrying forever with that same expired token, never able to recover — and separately, the
  background task meant to proactively refresh the token before expiry had no error handling at
  all, so a single transient failure could silently disable it for good. Both fixed in
  `roombapy-prime` v0.1.11a12; this alpha's pin bump brings the fix in directly.

Whether this is confirmed to be the exact mechanism behind every possible "lost connection"
report isn't certain — but both are real, independently-justified fixes now included regardless.

## 🔍 Under the hood

- Onboarding is a little faster: the config flow's own validation login is now reused for the
  immediate first setup instead of the two currently-redundant login sequences that alpha
  required before — single-use, short-lived, in-memory only.
- A real test-suite gap fixed alongside this: a config-flow test helper that bare-constructs a
  flow object (bypassing `__init__`, a deliberate pattern for keeping HA's own flow-registration
  machinery out of unit tests) hadn't been updated when the login-bridge attribute above was
  added, and was silently missing it — caught and fixed before this release, not after.

## ⚠️ Still known limitations, unchanged from v4.0.0a0

- **No battery percentage or docked/charging boolean yet.** Real progress upstream since the
  last alpha — the actual data location has been found (a previously-unknown, read-only cloud
  data source reports battery percentage and related fields, confirmed live against a real
  device) — but only the key *names* are confirmed so far, not the real values, and nothing is
  wired into an entity in this integration yet. A concrete next step, not a dead end.
- **"Find my robot" / locate remains broken.** Two independent, reasoned mechanisms have now
  been tried against a real device and confirmed NOT to work; a third, more promising candidate
  exists but hasn't been conclusively tested yet (a connectivity issue interfered with the one
  attempt so far). Still unresolved as of this alpha.
- **Region/zone cleaning**: still out of scope for this alpha.
- Please keep reporting anything unexpected — this remains an early alpha.

## Installation

Update through HACS as usual. The corrected `roombapy-prime` pin will be pulled in
automatically — no separate action needed.
