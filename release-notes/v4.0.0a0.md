# Roomba+ v4.0.0a0

> 4,028 tests passing (full suite, zero exclusions) · First alpha of V4/Prime support (cloud-only robots on iRobot's new-generation protocol) · Config Entry version unchanged (no migration needed — this is an entirely new, separate onboarding path, not a change to existing entries)
>
> This is a genuine alpha: it works against real hardware (multiple field testers, multiple accounts), but battery level and a direct docked/charging boolean are still unconfirmed and not yet exposed. If that matters for your use case today, you may want to wait for a later alpha.

## ✨ New: V4/Prime support (cloud-only robots)

iRobot's newer-generation robots (Combo/Plus 400-series and others on the "Prime" cloud
protocol) don't speak the local MQTT protocol this integration has always used — they're
cloud-only. This release adds a complete, separate path for them, built on the companion
[roombapy-prime](https://github.com/johnnyh1975/roombapy-prime) library (pinned to v0.1.11a6).

### Onboarding

- New third option in the initial setup flow, alongside "robot(s) found locally" and
  "set up manually": sign in with your iRobot cloud account.
- Prime robots on that account are set up directly. Classic robots on the *same* account are
  automatically completed too — a local network scan finds them by BLID, with a manual-IP
  fallback and an opt-in checkbox for also enabling cloud features on them (on by default).

### Vacuum control

- Start, pause, stop, dock, and locate all work, via the same command channel independently
  confirmed live against real devices.
- Live activity (cleaning/paused/docked/returning/idle) is derived from the robot's real-time
  mission event stream — 12 distinct event types mapped, 6 of them (start/reloc/travel/room/
  pause/fin) confirmed on the live push channel itself, 5 more via the historical mission-
  history endpoint, one (error) included defensively though never observed in any real capture
  yet. Room-in-progress, area cleaned, and pass count are exposed as attributes when available.
- Region/zone cleaning is out of scope for this alpha — requesting it gives a clear
  "not yet supported" message rather than silently doing nothing.

### Diagnostics

- Two new sensors: current mission event (the raw event type, for anyone who wants more
  granularity than the vacuum's own activity state) and connection health (whether the
  live data stream is currently working).

## ⚠️ Known limitations

- **No battery percentage or docked/charging boolean yet.** The activity states above are a
  reasoned approximation from mission events, not a full status readout. Multiple live field
  tests — including one watching a full 60 seconds after the robot was sent to dock — haven't
  turned it up yet; a longer post-dock observation window is the next thing being tried.
- **Region/zone cleaning**: not implemented for Prime robots in this alpha.
- **Mixed Classic+Prime households**: each robot is independently confirmed working, but
  running both types on one Home Assistant instance simultaneously hasn't been specifically
  tested yet.
- Everything above has been exercised against multiple real robots and separate accounts
  during development, but this is still an early alpha — please report anything unexpected.

## 🔍 Under the hood

- Entirely separate setup/coordinator/entity code paths for Prime robots — nothing about how
  existing Classic (local MQTT) robots work has changed, and no existing config entry needs
  any migration.
- Live data arrives via push (no polling) and reconnects automatically across network drops.
- The `roombapy-prime` dependency is now pinned to a specific tag (`v0.1.11a6`) in both
  `manifest.json` and `requirements-test-frozen.txt` — previously unpinned in the manifest and
  entirely absent from the frozen test requirements, meaning CI would likely have failed to
  properly test the V4/Prime code paths at all. Caught and fixed before this release, not after.
- Fixing the above surfaced a genuine follow-on conflict: `requirements-test-frozen.txt` had
  `paho-mqtt==1.6.1` hard-pinned (a frozen snapshot from before `roombapy-prime` was ever
  added), which is incompatible with `roombapy-prime`'s own `paho-mqtt>=2.0,<3.0` requirement
  (needed for paho-mqtt 2.x's `CallbackAPIVersion` API specifically). Resolved by bumping the
  pin to `paho-mqtt==2.1.0` — checked directly, not assumed: `roombapy` (the Classic library)
  itself declares compatibility up to `<3.0.0`, so this isn't a workaround, it's the actual
  correct shared version. Verified both by a clean `pip install --dry-run` resolution and by
  re-running the full test suite with 2.1.0 actually installed.
- A real CI failure caught before merge: hassfest rejected the pinned `manifest.json`
  requirement with `[REQUIREMENTS] Requirement "..." contains a space` — Home Assistant's own
  developer docs recommend the PEP 508 `"package @ git+https://..."` syntax for a git-based
  requirement, but hassfest's validator rejects exactly that (a known, documented
  inconsistency between HA's own docs and its own tooling — see home-assistant/core#123339 and
  #149833). Fixed by dropping the space immediately around the package-name/`@` separator
  (`roombapy-prime@git+https://...`), keeping the `@<tag>` pin at the end untouched — verified
  both against the documented community workaround and with a fresh `pip install --dry-run`.
  New `test_manifest_requirements.py` guards against this specific regression class going
  forward, since hassfest itself isn't runnable in this project's own local test environment.
- See `ROOMBA_PLUS_VERSION_PLAN_v4_onwards.md` in the repo for the full architecture writeup
  and the detailed evidence trail behind every confirmed piece of the protocol.

## ⏳ Still waiting on

Real-world confirmation from field testers on where battery/docked status actually lives on
the wire — this alpha ships with what's confirmed so far (room/mission progress, live
activity) rather than waiting for that to resolve. A live map-streaming mechanism was also
found during testing (the robot already pushes live map updates in a format this project's
tooling can already download and parse) but isn't wired into any entity yet — a plausible
follow-up for a future alpha, not part of this one's scope.
