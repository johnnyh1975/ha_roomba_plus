# Roomba+ v3.4.1 — Error Catalogue Expansion, Translations, and Mission Map Rotation

> 3,827 tests passing (+24) · Config Entry version 25 (no migration needed) · HACS: update as usual

A follow-up release built directly from field feedback and a fresh pass over
iRobot's own app: 32 new error codes (including a whole previously-unknown
Combo dock error category), error text now available in five additional
languages, and an optional rotation parameter for mission map images — plus
two long-standing open questions finally answered by field testers.

## ✨ New

### 32 new entries in the error catalogue

Direct analysis of iRobot's own Home app turned up error codes this
integration didn't recognise yet: nine individual codes (78, 79 — wheel
errors; 85, 86 — blocked path to the charging station; 91–93 — workspace
path errors requiring a map retrain; 98 — software error; 99 — navigation
problem), plus a full 23-code category (450–463, 501–509) for Combo
wet-mopping tank/dock errors that wasn't represented at all before. That
category has a consistent, useful pattern worth knowing: instead of failing
the mission outright, these all mean the robot switched to vacuum-only and
kept going.

Every code was checked against the app's actual displayed text before being
added — an earlier pass at this list had mistakenly proposed about twenty
codes that turned out to belong to iRobot's Terra lawn-mower line, which
happens to share the same app and error-code numbering. Those were caught
and excluded before release.

### Error messages in five more languages

`last_error_code`'s description and recommended action, and the recurring-error
repair issue, are now available in German, French, Italian, Spanish,
Portuguese, and Dutch, not just English — matching whatever language your
Home Assistant instance is set to. German has been reviewed directly;
the other four are machine-translated and offered with that caveat — if
you spot something that reads awkwardly in your language, an issue report
with the correction is very welcome. Falls back to English automatically
for any language not covered, and per field if only part of a translation
exists, so you'll never see a blank recommendation.

### Rotate mission map images — `?rotate=` on `.../map.png`

Requested by a field tester whose Smart Map's stored orientation didn't
match their dashboard layout: the mission map PNG endpoint now accepts an
optional `rotate` query parameter (`90`, `180`, or `270`, clockwise), e.g.

```
/api/roomba_plus/{entry_id}/missions/latest/map.png?rotate=270
```

Useful directly in picture-card and notification image URLs — no external
image processing needed. Lossless: an exact pixel rearrangement, not an
interpolated rotation, so there's no quality loss at these fixed angles.
Omit it, or send anything else, and you get exactly the same unrotated
image as before — this is purely additive.

## 🔍 Confirmed by field testers

**Mission cleaning maps work on i-series lewis firmware.** Previously only
confirmed on Braava jet m6 (sapphire firmware); a field test on an i7
returned a valid, correctly-populated map image, closing this open
question from v3.3.0.

**The room-type suggestion feature holds up on a second, independent
device.** `region_suggestions` (the data behind the automatic room-name
suggestion introduced in v3.2.0) had only ever been checked against one
sample. A second sample, from a different robot and firmware generation,
matched the existing logic's expectations exactly — including robots
where the top suggestion should be rejected (a negative score) in favour
of a lower-but-positive one.

## 🛠 Fixed

Two issues found while finishing the translation work above, before
release, not in the field:

- A missing import meant a translated-error lookup would have raised
  immediately the first time a recurring-error repair issue was created
  on a non-English system.
- An unrecognised error code returned an empty result for English, but
  three empty-string fields for every other language — a subtle
  difference that would have silently defeated the fallback text
  ("Error {code}") a caller further up relies on when a code truly isn't
  in the catalogue.

Two more, found from a field report (i3+, thank you mdarocha) after an
already-used robot was added to Roomba+ for the first time:

- **No map, and an implausibly low "total cleaned area."** Some robots
  report persistent Smart Maps without also reporting the live-pose
  capability flag map setup previously required as a prerequisite — a
  gap the existing lewis-firmware cloud fallback (v3.4.0) didn't close,
  since it never got the chance to engage. Map setup, and the cloud
  connection it also unlocked, now activate correctly from real map
  data alone when live pose isn't reported. If your map has been coming
  up empty and "total cleaned area" looked far too small compared to
  the official app, this is very likely why.
- **Maintenance countdowns showing "0h remaining" immediately after
  first setup on an already-used robot.** Without ever knowing when you
  last actually replaced a filter or brush, the countdown previously
  assumed the robot's *entire* prior lifetime counted against the
  interval — a robot with months of history via the official app would
  show up as instantly, alarmingly overdue. It now assumes maintenance
  is current as of whenever Roomba+ starts watching, the same reasonable
  assumption you'd make yourself on day one; nothing changes once you
  press a reset button for real.

## 📦 Upgrade notes

No breaking changes, no storage migration, no configuration required.
Existing `last_error_code` sensors and repair issues will simply start
showing translated text if your Home Assistant language isn't English.
The `rotate` parameter is opt-in — add it to a picture-card or
notification's image URL wherever your Smart Map needs it; nothing
changes for anyone who doesn't. The two maintenance/map fixes apply
automatically on next restart — no action needed, and no effect on any
installation that was already working correctly.
